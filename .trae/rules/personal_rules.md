# 个人工作规则 - 精简版

## 🚨 核心原则
**任何代码变更都必须经过完整的测试验证流程，绝对不能跳过！**

## 修改前3步
1. **理解功能** - 打开 `feature_list.md`，找到要修改的功能
2. **识别影响** - 分析会影响哪些功能，哪些功能依赖它
3. **创建计划** - 明确要改什么、关联什么、测试什么

## 修改时3条规则
1. ✅ 一次只改一个地方，完成测试后再改下一个
2. ✅ 保留原始代码（先注释，确认正常后再删除）
3. ✅ 不要修改不理解的代码（先问清楚再改）

## 修改后4阶段测试

### 阶段1：基础测试
- [ ] 应用能正常启动，无编译/运行时错误
- [ ] 被修改的功能正常工作（正常情况+边界情况）
- [ ] 相关功能没有被破坏

### 阶段2：完整测试
- [ ] WebSocket 连接/接收消息
- [ ] 消息去重功能
- [ ] 重点消息检测+弹窗
- [ ] 复制功能
- [ ] 窗口置顶+状态记忆
- [ ] 系统托盘

### 阶段3：深度检查
- [ ] 打开开发者工具，无红色错误
- [ ] 查看Rust日志，无异常
- [ ] 长时间运行，内存稳定

### 阶段4：最终验证
- [ ] 关闭应用，重新打开
- [ ] 应用能正常启动
- [ ] 之前的设置被正确保存
- [ ] 所有功能仍然正常

## 绝对禁止的5件事
1. ❌ 禁止说"应该没问题" - 必须实际测试
2. ❌ 禁止只测试修改的功能 - 必须测试相关功能
3. ❌ 禁止跳过验证步骤 - 每个步骤都必须完成
4. ❌ 禁止同时修改多个功能 - 一次只改一个
5. ❌ 禁止在不理解的情况下修改 - 先理解再修改

## 代码修改4个禁止
1. ❌ 不要删除不确定的代码（先注释后删除）
2. ❌ 不要修改全局变量名（除非知道所有使用点）
3. ❌ 不要修改事件名称（除非知道所有监听点）
4. ❌ 不要修改函数签名（除非知道所有调用点）

## 发现问题怎么办
1. 立即停止修改
2. 恢复到修改前的状态
3. 重新分析问题
4. 逐步修改，逐步测试

## 验证完成的标准
只有满足以下条件，才能说修改完成：
- ✅ 功能验证：修改的功能正常，相关功能正常，无新错误
- ✅ 测试验证：所有测试用例通过，手动测试完成，长时间运行测试通过
- ✅ 代码质量：无语法错误，无明显逻辑错误，代码风格一致

## 记住
**宁可多花时间测试，也不要花时间修复bug！**
**测试不充分 = 一定会出问题！**
**现在多花10分钟测试，将来少花1小时修复！**
